# Архитектура Android-приложения XPump

**XPump** – это фитнес-трекер с геймификацией и системой спортивных нормативов, рассчитанный на будущий рост функционала (многопользовательские режимы, облачная синхронизация, push-уведомления, аналитика и пр.). Для обеспечения **масштабируемости, поддерживаемости и чистоты кода** используется современная архитектура **MVVM + Repository**, а также принципы Clean Architecture (чистой архитектуры) с разделением на слои **Presentation**, **Domain** и **Data**. Такая структура облегчает добавление новых возможностей, тестирование и сопровождение приложения.

- **Модуль Presentation (UI)** – **главный Android-модуль приложения**. Содержит весь код, связанный с интерфейсом и взаимодействием с пользователем. Здесь находятся `Activity` и `Fragment` для экрана(ов) приложения, а также соответствующие им `ViewModel` и UI-слои. В XPump планируется архитектура с **одной Activity**, которая играет роль контейнера для навигации, и несколькими Fragment для разных экранов. Такой подход («single-activity, multi-fragment») официально рекомендован Google для современных приложений.

    - **Activity**: будет единственной (например, `MainActivity`) и выступит хостом для фрагментов и навигационного графа. В ней инициализируется навигация (Jetpack Navigation Component) и глобальные UI-компоненты (например, нижняя навигация или Drawer, если требуется). Activity **не содержит бизнес-логики или данных**, она лишь отображает фрагменты и обрабатывает системные события (как контейнер).

    - **Fragment**: отвечает за отрисовку экрана (определённой функции приложения) и обработку пользовательского ввода на этом экране. Для каждого ключевого экрана XPump будет создан свой Fragment. Например: `ProfileFragment` (профиль пользователя с уровнем и нормативами), `WorkoutHistoryFragment` (лента истории тренировок), `ExerciseListFragment` (библиотека упражнений), `ProgramListFragment` (список тренировочных программ), `WorkoutSessionFragment` (экран проведения тренировки, ввода подходов/весов) и т.д. Каждый Fragment связан с собственным `ViewModel`. Фрагмент через ViewBinding получает ссылки на View-элементы и подписывается на данные из ViewModel (LiveData/StateFlow) для отображения.

    - **ViewModel**: находится в слое презентации, но отделён от UI. В каждом ViewModel содержится логика представления: получение данных из domain-слоя (через UseCase или напрямую из Repository), подготовка данных для отображения, обработка событий UI (нажатия кнопок и т.п.). ViewModel **хранит состояние экрана** (например, список упражнений, текущий прогресс загрузки, ошибки) в виде LiveData или (предпочтительно) `StateFlow`. Использование **StateFlow** в связке с корутинами обеспечивает более эффективное и современное обновление UI, избегая лишних затрат ресурсов по сравнению с LiveData. ViewModel не содержит ссылок на View и живёт дольше фрагмента, сохраняя данные при поворотах экрана. В XPump ViewModel’ы помечаются аннотацией `@HiltViewModel` и получают зависимости через конструктор (DI) – например, UseCase или Repository-интерфейсы.

- **Модуль Domain (бизнес-логика)** – чисто Kotlin-библиотека (без Android-зависимостей), которая реализует бизнес-логику приложения. **Domain-слой** включает в себя описание основных **сущностей (Entity или модели домена)**, а также **интерфейсы репозиториев** и **UseCase/Interactor** для реализации сценариев использования. Этот слой отвечает за правила и операции приложения, оставаясь изолированным от деталей реализации (например, от конкретной БД или фреймворков). Благодаря этому, domain-код легко тестировать (можно запускать unit-тесты без эмулятора). Основные компоненты domain-слоя XPump:

    - **Entity (Domain Model)**: классы, описывающие ключевые сущности предметной области, с которыми работает логика. Примеры: `Exercise` (упражнение: id, название, группа мышц, описание, изображение), `Workout` (тренировка: id, дата, список выполненных подходов), `WorkoutSet` (отдельный подход: ссылка на упражнение, вес, количество повторений), `Program` (тренировочная программа: id, название, описание, требования к уровню), `UserProfile` (профиль пользователя: текущий уровень, опыт, весовая категория, текущие рекорды по упражнениям). Эти domain-модели используются в приложении и **независимы от деталей хранения** (то есть не обязательно 1-в-1 соответствуют таблицам БД).

    - **Repository (интерфейсы)**: определяют **контракты доступа к данным** для каждой группы сущностей. Например, `WorkoutRepository` с методами типа `getAllWorkouts()`, `addWorkout(workout)`, `getWorkoutById(id)`; `ExerciseRepository` с методами `getExercises()`, `getExerciseById()`; `ProgramRepository` (`getAvailablePrograms()` и т.д.); `UserRepository` (`getUserProfile()`, `updateUserProfile()` для уровня и опыта). **Важно:** Repository в domain – это только интерфейсы, не содержащие реализации. Они описывают, какие данные нужны приложению, абстрагируясь от того, откуда эти данные берутся (локально или из сети). Это и есть применение принципа **инверсии зависимостей** – верхний слой зависит не от конкретной реализации, а от абстракции (интерфейса).

    - **UseCase / Interactor**: классы, реализующие **бизнес-сценарии** (варианты использования) путём композиции операций над сущностями. Каждый UseCase, как правило, выполняет одну законченную операцию, которая может включать обращение сразу к нескольким репозиториям и применяет бизнес-правила. Например, в XPump можно выделить такие use-case: `LogWorkoutUseCase` (сохранить выполненную тренировку: добавить запись в историю, обновить опыт пользователя, пересчитать прогресс норматива), `GetWorkoutHistoryUseCase` (получить список всех тренировок из репозитория), `GetProfileOverviewUseCase` (загрузить профиль пользователя и вычислить текущий прогресс до следующего спортивного разряда), `AwardExperienceUseCase` (начислить XP и при необходимости повысить уровень), `GetExercisesUseCase` (выгрузить библиотеку упражнений), `GetProgramsUseCase` (получить список программ, пометив, какие разблокированы для текущего уровня) и др. UseCase вызывает методы репозиториев и выполняет логику (например, фильтрацию, сортировку, расчёты нормативов). Они возвращают результат обратно во ViewModel. **Применение слоя UseCase необязательно для простых операций**, но очень полезно для концентрации сложной логики (например, расчёт спортивного разряда по результатам упражнений и весовой категории пользователя). Такой подход защищает UI-слой от изменений в логике и API данных и облегчает повторное использование логики в разных частях приложения.

- **Модуль Data (слой данных)** – отвечает за управление данными: хранение их локально, загрузку с сервера (в перспективе), кэширование и поставку в приложение. В MVP XPump Data-слой обеспечивает **offline-first** работу – все основные функции работают без интернета, используя локальную базу данных. **Data-модуль** зависит от Android и сторонних библиотек (Room, SQLite, возможно Retrofit/Firebase в будущем), реализуя интерфейсы репозиториев из domain. Структура data-слоя:

    - **Room Database**: локальная база данных SQLite, управляемая через библиотеку Room. В Data-модуле будет описан класс БД (например, `XPumpDatabase`) с аннотацией `@Database`, который включает DAO для сущностей. Будут созданы **Entity-классы** с аннотациями `@Entity` для таблиц: `WorkoutEntity`, `ExerciseEntity`, `ProgramEntity`, `WorkoutSetEntity` и т.п. (соответствующие domain-моделям). Эти классы, помеченные аннотациями, определяют схему БД. Если domain-модели отличаются по структуре, могут быть введены отдельные Data Models (DTO) и **мэпперы** для преобразования их в доменные сущности. Однако на этапе MVP можно упростить и использовать сами Entity как доменные модели, чтобы не дублировать код, учитывая, что приложение локальное.

    - **DAO (Data Access Object)**: интерфейсы с аннотациями `@Dao`, определяющие методы для выполнения запросов к базе (CRUD-операции). Например, `WorkoutDao` с методами `@Query("SELECT * FROM workouts") fun getAll(): Flow<List<WorkoutEntity>>`, `@Insert fun insertWorkout(workout: WorkoutEntity)`, `@Transaction fun insertWorkoutWithSets(...)` для комплексных операций; `ExerciseDao` (`getAllExercises()` и т.п.), `ProgramDao`, `UserDao` (если профиль хранится частично в БД). **Room автоматически генерирует реализацию DAO** на основе этих интерфейсов, поэтому разработчику нужно писать только сигнатуры методов и SQL-запросы. DAO будут возвращать _Flow_ или LiveData, чтобы можно было наблюдать за изменениями данных в реальном времени.

    - **Репозитории (реализация)**: класс(ы), реализующие интерфейсы из domain-слоя, используя DAO и другие источники данных. Например, `WorkoutRepositoryImpl : WorkoutRepository`, который внутри обращается к `WorkoutDao` для получения/сохранения тренировок. **Repository является посредником между базой данных (и другими источниками) и ViewModel**. В будущем, здесь же будет логика объединения данных из локального и удаленного источников. Например, `WorkoutRepositoryImpl` сможет обращаться к локальной БД **и** к удалённому API (Firebase) – под капотом решая, когда обновить данные. На этапе MVP репозитории обращаются только к локальной Room-БД и к SharedPreferences.

        - Для примера, `ExerciseRepositoryImpl` может при первом запуске заполнить базу данных предустановленным списком упражнений (например, из заранее подготовленного JSON в ресурсах) и затем выдавать список через `exerciseDao.getAllExercises()`. `ProgramRepositoryImpl` – выдавать список программ, помечая поле `isUnlocked` в каждом на основе текущего уровня пользователя (который можно получить из `UserRepository`). Такая логика может быть реализована либо в самом репозитории, либо в соответствующем UseCase.

        - **Принцип Single Source of Truth (SSOT)**: Репозиторий гарантирует, что каждый тип данных имеет одно достоверное хранилище. Для XPump таким источником будет локальная база (offline-first). Например, истории тренировок хранятся только в Room, и UI получает их только из Room (через репозиторий) – даже если в будущем данные синхронизируются с облаком, локальная БД останется источником истины, а сеть – лишь для обновлений. Такой подход упрощает работу в офлайн-режиме: приложение может работать **без сети**, используя последние сохранённые данные.

    - **SharedPreferences (либо DataStore)**: для хранения настроек и некоторых данных профиля (уровень, опыт пользователя, возможно выбранная весовая категория) используется хранилище ключ-значение. В MVP применяются `SharedPreferences` (как указано в требованиях), хотя современным решением является Jetpack DataStore, предлагающий асинхронную работу на основе Flow. В коде будет создан класс-обёртка, например `PreferencesStorage` или `UserPreferencesRepository`, который инкапсулирует доступ к `SharedPreferences` API (или DataStore) и предоставляет typed-методы, например: `getUserLevel(): Flow<Int>`, `setUserLevel(level: Int)`, `getWeightCategory()` и т.д. Это позволит легко заменить реализацию на DataStore в будущем без изменения остального кода.


**Иерархия пакетов** в каждом модуле будет организована для удобства поддержки. Один из подходов – **Feature-based** (по функциональным экранам). Например, в модуле Presentation пакет `ui.workout` содержит `WorkoutSessionFragment` и `WorkoutViewModel`, пакет `ui.history` – `WorkoutHistoryFragment` и `HistoryViewModel`, `ui.profile` – `ProfileFragment`/`ProfileViewModel`, `ui.exercises` – экран списка упражнений, и т.д. Это логически группирует связанные компоненты. Внутри domain-модуля можно завести пакеты по сущностям или функционалу: например, `domain.workout` (модели Workout/Set, интерфейс WorkoutRepository, use-case LogWorkout и GetHistory), `domain.profile` (модель UserProfile, репозиторий/преференсы, use-case AwardExperience, ComputeRank и т.д.). В data-модуле – пакеты `db` (с entity и dao), `repository` (с реализациями репозиториев), `datasource` (если выделять локальный/удалённый источники отдельно), `preferences` (для работы с SharedPrefs). Такая структура снижает связность между модулями и частями приложения, облегчает навигацию по проекту и масштабирование команды разработчиков.

## Архитектурные паттерны и лучшие практики

При реализации XPump используются проверенные паттерны архитектуры и инструменты, обеспечивающие чистый и поддерживаемый код:

- **MVVM (Model-View-ViewModel)** – основной паттерн презентационного слоя. Он разграничивает **View (UI)** и **ViewModel (логика презентации)**, что уменьшает **зацепление (coupling)** и упрощает масштабирование приложения. `Activity/Fragment` (View) наблюдает за состоянием из `ViewModel` и отображает его, а пользовательские действия (нажатия кнопок) проксируются во ViewModel (например, через вызовы методов или `SharedFlow` событий). ViewModel же обращается к модели (UseCase/Repository) для изменения данных. Такой односторонний поток данных (UDF – unidirectional data flow) упрощает отслеживание изменений и уменьшает количество ошибок: данные текут от нижних слоёв к UI, а события – от UI вниз к слою данных. В MVVM **рекомендуется максимально “тонкий” UI**, без бизнес-логики. Эта парадигма поддерживается Jetpack Component’ами: `ViewModel`, `LiveData/StateFlow` и обеспечивает устойчивость к поворотам экрана и прерываниям жизненного цикла Activity.

- **Repository Pattern (паттерн репозиторий)** – слой абстракции между источниками данных и остальной частью приложения. Как описано выше, Repository предоставляет простой API для ViewModel и UseCase, скрывая детали получения данных. Например, `historyViewModel` вызывает `workoutRepository.getAllWorkouts()` и получает сразу готовый список (Flow) из локальной БД, не беспокоясь, из какого источника эти данные пришли. Внутри репозиторий мог объединять несколько источников (например, кэш + сеть) – для XPump это станет актуально, когда появится облачная синхронизация. Согласно рекомендациям Google, каждый **репозиторий может комбинировать локальный и удалённый источник** данных, причём локальный действует как **источник истины** в offline-first приложении. Мы закладываем эту возможность с самого начала: например, `WorkoutRepositoryImpl` после добавления тренировки в Room может отправлять данные на сервер (если онлайн) или ставить задачу синхронизации через WorkManager. В MVP реализация будет проще – только локальные операции, но архитектура готова для расширения. Репозиторий также может выполнять **мэппинг** между различными моделями (DTO -> Domain), чтобы UI не зависел от формата хранения.

- **Чистая архитектура (Clean Architecture)** и **слоистость**: Проект строго делится на слои ответственности (UI / Domain / Data). Благодаря этому достигается **слабая связность** и изоляция изменений – изменения в нижнем слое (например, замена SQLite на другую БД) не затрагивают напрямую верхние слои, если сохраняется контракт интерфейсов. Каждый слой общается с соседними через четко определённые интерфейсы или data-классы, что соответствует принципу **Separation of Concerns** (разделения ответственности). Кроме того, соблюдается принцип **инверсии зависимостей**: зависимостями нижних слоёв управляет верхний (например, ViewModel получает реализацию репозитория через конструктор, а не создаёт её сам). Мы используем Hilt DI (см. ниже) для автоматизации этого процесса. Clean Architecture также способствует тестируемости и гибкости к изменениям функционала. В частности, domain-слой не зависит от платформы, и его можно легко покрыть юнит-тестами, а UI-слой не “знает” подробностей про данные, что упрощает изменение источников данных или логики без переписывания интерфейса.

- **Dependency Injection (DI)** – для внедрения зависимостей применяется **Hilt (Dagger Hilt)**, что является официально рекомендуемым подходом Google для Android-проектов. DI позволяет отделить создание объектов от их использования: специальные классы-«контейнеры» поставляют экземпляры ViewModel, репозиториев, DAO и т.д., а потребители просто объявляют их в конструкторах. **Hilt** существенно снижает шаблонный код, автоматически создавая и управляя графом зависимостей приложения. В XPump будет создан класс приложения `XPumpApp` с аннотацией `@HiltAndroidApp` (для генерации базового компонента). Каждый Android-компонент (Activity, Fragment) помечается `@AndroidEntryPoint` для поддержки внедрения. Модули Hilt (`@Module`) предоставят зависимости: например, `DatabaseModule` с функцией `@Provides fun provideDatabase(...)`, `DaoModule` (`provideWorkoutDao` и т.д.), `RepositoryModule` (`provideWorkoutRepository(workoutDao) : WorkoutRepository` и аналогично для других репозиториев). Hilt автоматически сгенерирует код, который при старте приложения создаст синглтоны БД и репозиториев и будет прокидывать их в нужные места. В ViewModel мы используем аннотацию `@HiltViewModel` и конструктор с `@Inject`, чтобы Hilt мог создавать экземпляры моделей представления со всеми зависимостями (например, UseCase). DI обеспечивает **слабо связанный код** и облегчает подмену реализаций (например, на моки в тестах), а также автоматически учитывает жизненный цикл Android-компонентов (Hilt умеет создавать зависимости, скоупленные к Activity, Fragment и пр., освобождая их память при уничтожении компонентов). Это избавляет от ручного управления синглтонами и передачей контекстов. В будущем через DI будет нетрудно добавить новые сервисы – например, клиент для удалённого API, который также предоставляется через Hilt-модуль и внедряется в нужные репозитории.

- **Навигация**: используется **Jetpack Navigation Component** для определения экранов и переходов между ними. Навигация описывается в навигационном графе `nav_graph.xml`, где будут объявлены все Fragment-экраны приложения XPump и возможные переходы (Actions). Например, из `ProfileFragment` можно навигировать к `ExerciseListFragment` (кнопка «Библиотека упражнений») или к экрану настроек; из `ProgramListFragment` – к экрану выполнения тренировки (передавая id выбранной программы); из `HistoryFragment` – к деталям конкретной тренировки (если планируется такой экран). Navigation Component поддерживает передачу параметров между фрагментами с помощью **Safe Args (Gradle-плагин)** – это избавляет от необходимости вручную сериализовать данные в Bundle. Для XPump Safe Args пригодится, например, чтобы передать идентификатор выбранной программы на экран тренировки, либо объект упражнения на экран его описания. Мы планируем применить **Single-Activity подход** с хостингом всех фрагментов в `MainActivity`, что упрощает глобальную навигацию и обмен данными между экранами. При необходимости разделения по флоу (например, отдельный стек навигации для режима авторизации в будущем) можно использовать несколько графов или вложенные NavHost, но пока достаточно одного. Navigation Component поможет также удобно обработать Up-навигацию и кнопки Back (он автоматически синхронизируется с `OnBackPressed` колбеком). В рамках best practices, все переходы будут описаны декларативно в графе, а не через прямые вызовы FragmentManager в коде – это повышает наглядность структуры приложения.

- **Работа с Room**: для доступа к базе данных используем **корутины и Flow**. DAO-функции будут помечены `suspend` (для операций вставки, обновления) или возвращать `Flow<List<Entity>>`/`LiveData` (для чтения). Это позволяет вызывать БД-операции без блокировки основного потока (Room под капотом поддерживает асинхронность для Flow/LiveData). ViewModel будет вызывать репозиторий, который обращается к DAO. Например, `workoutRepository.getAllWorkouts()` может возвращать Flow, на который ViewModel подпишется (преобразовав, при необходимости, в StateFlow или LiveData для UI). При добавлении новой тренировки `insertWorkout()` в DAO будет выполнен на IO-диспетчере (корутина), и Room автоматически уведомит Flow `getAllWorkouts()` о новом значении – в итоге HistoryFragment получит обновлённый список автоматически. **Таким образом достигается реактивность интерфейса**, без ручного оповещения – UI всегда отражает актуальные данные. Здесь проявляется плюс **StateFlow + Room**: при изменении данных в базе, привязанный StateFlow во ViewModel обновит состояние экрана без дополнительных усилий (набор состоит из Flow->StateFlow в VM и collect в Fragment).

- **UI слой и лучшая практика**:

    - Используем **XML-верстку** с **ViewBinding**. Для каждого layout-файла генерируется класс Binding, и во фрагменте мы безопасно получаем ссылки на View через него. Это исключает риски `NullPointerException` и повышает читаемость (например, `binding.textView.text = ...`). ViewBinding подключается в Gradle скрипте модуля presentation.

    - **Разделение ответственности в UI**: Фрагменты будут максимально простыми: подписка на данные ViewModel и рендеринг, отправка событий (типа нажатий) обратно в ViewModel. Для более сложных экранов (например, экран выполнения тренировки, где можно динамически добавлять подходы) можно вынести логику управления UI-компонентами в отдельные вспомогательные классы или использовать **Custom View**. Например, реализовать кастомный компонент `ProgressBarWithLabel` для отображения прогресса до следующего разряда (чтобы переиспользовать его для разных упражнений на экране профиля).

    - **Adapter & RecyclerView**: для списков (история тренировок, список упражнений, программ) будут созданы адаптеры на базе RecyclerView. Рекомендуется использовать `ListAdapter` с **DiffUtil** для эффективного обновления списков при изменении данных – это минимизирует перерисовку элементов. Например, `WorkoutHistoryAdapter` для ленты истории, `ExerciseAdapter` для библиотеки упражнений. Элементы списка могут иметь собственные ViewModel или воспользоваться простым биндингом данных в методе `onBind`.

    - **Материальный дизайн и навигация**: для основного меню возможно использование BottomNavigationView (если предусмотрены вкладки, например: Профиль, Тренировки, Программы, Упражнения). Этот BottomNav можно разместить в Activity и связать с NavController (через `NavigationUI.setupWithNavController`), чтобы переключение между разделами было удобным. Для настроек можно использовать PreferenceFragmentCompat, либо простой отдельный экран с переключателями, значения которых хранятся в SharedPreferences.

- **Обработка жизненного цикла и состояний**: ViewModel будет хранить состояние экрана, например, через отдельные data-классы – **UI State**. Это хороший паттерн: определить data class, содержащий все данные, необходимые для отрисовки экрана (список элементов, флаги загрузки, ошибки и т.д.), и держать экземпляр этого класса в `StateFlow`. Тогда Fragment, подписываясь, получает всегда полный снэпшот состояния и обновляет UI. Например, `HistoryViewState(val workouts: List<Workout>, val isLoading: Boolean, val error: String?)`. Меняя `StateFlow` в ViewModel (copy копиями), мы гарантируем иммутабельность данных в UI. Кроме того, события одноразовые (типа "показать Toast") можно обрабатывать через `SharedFlow` или EventWrapper, чтобы не потерять при повороте экрана. Эти подходы относятся к best practices по управлению состоянием UI в MVVM.

- **Безопасность данных и ресурсов**: хотя MVP – локальный, мы учтём моменты сохранения данных. Например, для критичных данных (история тренировок) можно настроить **бэкап** (Android Auto Backup или вручную экспорт/импорт, если актуально) – но это вне рамок MVP. Если будем хранить личные данные (например, вес пользователя, лучшая попытка) – в будущем стоит рассмотреть **Encryption (EncryptedSharedPreferences или шифрование базы)**, особенно перед облачной синхронизацией. Эти вопросы можно учитывать архитектурно: например, изолировать работу с SharedPreferences через один класс, чтобы потом его заменить на зашифрованную версию при необходимости.


## Поддержка масштабируемости и будущих функций

Архитектура XPump спроектирована с учётом **будущего расширения**. Благодаря модульности и слабой связанности добавление новых фич потребует минимальных изменений в существующем коде. Рассмотрим, как подготовленные решения помогут внедрить ожидаемые улучшения:

- **Авторизация пользователей**: В перспективе, приложение может получить систему аккаунтов (Firebase Auth или собственный сервер). Благодаря слою domain и абстракциям репозиториев, можно добавить, например, `AuthRepository` с методами `signIn(email, pass)`, `signOut()`, `getCurrentUser()` и т.п. В data-слое появится реализация, использующая Firebase Auth SDK. UI-слой может получить новый экран LoginFragment, а остальная часть приложения практически не затронется – сервисы синхронизации будут брать ID текущего пользователя из AuthRepository. Кроме того, можно будет разделить данные разных пользователей, храня их, например, в разных таблицах или разделах (либо очищая локальные данные при логауте). **Навигационно** можно оформить auth как отдельный nav-graph, включаемый до запуска основного приложения (Single-Activity подход это позволяет). С DI интеграция тоже проста: FirebaseAuth инжектится в AuthRepositoryImpl через Hilt. Таким образом, за счёт слоистой архитектуры авторизация добавляется без рефакторинга ядра приложения.

- **Синхронизация с облаком (онлайн-режим)**: В текущем offline-first подходе уже заложено, что локальная БД – источник истины. Для добавления облака (например, Firebase Firestore или REST API сервер) можно расширить репозитории. Один из вариантов – реализовать **паттерн репозитория с двумя источниками**: локальный и сетевой. При запросе данных репозиторий сначала выдаёт локальные данные (для быстрого отображения), а параллельно инициирует обновление из сети. После получения ответа – сохраняет/обновляет данные в локальной БД и тем самым UI автоматически обновится (благодаря Flow). Этот подход соответствует рекомендациям по offline-first синхронизации. Например, `WorkoutRepositoryImpl` сможет иметь ссылку на `RemoteDataSource` (класс, делающий сетевые вызовы, например через Retrofit или Firebase SDK) и на `LocalDataSource` (DAO). Добавление такого функционала означает:

    - Создание моделей для сети (DTO) и мэпперов к Domain-моделям.

    - Реализация методов в репозитории: например, `getAllWorkouts()` проверяет `if (networkAvailable) fetchFromRemoteAndUpdateLocal()` прежде чем вернуть Flow из локальной БД.

    - Использование WorkManager или других планировщиков для периодической синхронизации или фоновой отправки данных (например, отправлять на сервер новые тренировки, когда устройство онлайн).

    - Все эти дополнения **не потребуют изменений** в UI или ViewModel – они по-прежнему вызывают тот же метод репозитория. Меняется только «начинка» репозитория, что демонстрирует гибкость архитектуры.

    - **Конфликты и объединение данных**: если планируется двунаправленная синхронизация, можно будет внедрить стратегию merge-конфликтов в репозитории или в remote source.

    - **Push-уведомления**: интеграция FCM (Firebase Cloud Messaging) для рассылки уведомлений о новых программах или сообщениях также не нарушит архитектуру. Добавится сервис (наследник FirebaseMessagingService) в приложении, который в `onMessageReceived` может, например, обращаться к репозиторию для обновления данных или к NotificationManager для показа нативного уведомления. Этот сервис регистрируется в манифесте. Так как push-уведомления – отдельный аспект, их внедрение потребует лишь небольших точечных дополнений: получение FCM токена (можно через тот же AuthRepository, если нужно ассоциировать с пользователем), и обработчик сообщений. **Важное замечание**: push-уведомления и фоновые обновления данных должны учитывать актуальность данных: наша architecture уже готова к тому, чтобы обновлённые данные были помещены в локальную БД репозиторием, а UI сам обновится, если открыт соответствующий экран.

- **Аналитика**: Для отслеживания пользовательских действий и метрик (например, с помощью Firebase Analytics, AppMetrica или др.) достаточно встроить вызовы логирования в нужных местах приложения. Архитектура позволяет сделать это централизованно. Например, можно создать `AnalyticsTracker` (интерфейс) и его реализацию с вызовом Firebase Analytics SDK. Через DI внедрить его в нужные ViewModel или UseCase. Затем в коде, где происходит важное событие (пользователь завершил тренировку, получил новый уровень, открыл программу), вызвать `analytics.logEvent("WorkoutCompleted", params)`. Поскольку такой вызов не влияет на основную бизнес-логику, его добавление не сломает поведение. Можно также воспользоваться возможностями отслеживания экранов через Navigation (NavController взаимодействует с Analytics автоматически, если включено). Важное преимущество: все данные о действиях пользователя (например, список упражнений, который он просматривает) у нас есть в ViewModel, так что передать их в аналитику не составит труда. **Модульность** помогает при аналитике – можно сделать отдельный модуль `analytics` с обёрткой над SDK и подключать его при сборке релиза, не смешивая с бизнес-кодом.

- **Многопользовательский режим / PvP**: В проекте упомянут "многопользовательский", вероятно, подразумевается возможность конкуренции или совместных тренировок. Такое расширение обычно включает сетевое взаимодействие (таблицы лидеров, обмен данными с друзьями). С архитектурной точки зрения, это может быть реализовано добавлением новых **UseCase** (например, `GetLeaderboardUseCase`, `ChallengeFriendUseCase`) и соответствующих методов в репозиториях (которые теперь обзаведутся сетевым слоем). UI-слой получит новые экраны (например, рейтинг, профиль друга и т.п.). Благодаря тому, что ядро (история тренировок, профиль) уже отделено, многопользовательские функции будут добавлены параллельно, не ломающая старый функционал. Например, можно в существующий ProfileFragment добавить вкладку "Рейтинг" или кнопку "Поделиться результатом" – и эта кнопка будет вызывать новый UseCase, отправляющий данные на сервер.

    - **Разделение на фиче-модули**: при значительном росте функционала (например, отдельный модуль для соц. функционала) архитектура позволяет выделять **feature modules**. Уже сейчас проект можно заложить как мультимодульный: например, модуль `:featureHistory`, `:featureProfile`, `:featureWorkout`. В MVP с маленькой командой это избыточно, но в будущем, если над проектом будет работать много людей, можно физически разделить код по Gradle-модулям. Преимущества – ускорение сборки, параллельная работа над разными модулями, возможность переиспользовать модули (например, вынести `:core:database` модуль для других приложений). **Высокая Cohesion и низкая Coupling** принцип останется: каждый модуль реализует свою обязанность и минимально зависит от остальных.


Резюмируя, выбранная архитектура (MVVM + Repository + Clean Architecture принципы) обеспечивает **устойчивость к изменениям** и росту: добавление нового источника данных или функциональности потребует реализовать новый модуль/класс, но не менять существующие, что минимизирует риск регрессий. Разделение ответственности делает код проще для понимания и сопровождения: команда сможет работать над разными частями (UI, БД, сеть) почти независимо. В итоге, XPump готов к эволюции от локального MVP до полноценного облачного сервиса с минимальными архитектурными изменениями.

## Детальный план разработки MVP

Ниже представлен поэтапный план реализации MVP-версии XPump с указанием порядка работ, данных для хранения, а также рекомендаций по использованию автогенерации кода и тестированию:

1. **Инициализация проекта и настройка окружения**

    - **Структура модулей**: Создать проект в Android Studio, добавить модули: `app` (Android application module для UI), `domain` (Kotlin Library), `data` (Android Library, т.к. Room требует Android SDK). Можно воспользоваться шаблонами или добавить модули через Gradle вручную. Настроить в settings.gradle связь модулей. Добавить зависимость `app -> domain` и `app -> data`, а также `data -> domain` (т.к. реализации репозиториев будут использовать интерфейсы domain). Если планируется модуль `common` для утилит (например, расширения Kotlin, константы), подключить его в остальные.

    - **Gradle зависимости**: Подключить необходимые библиотеки в Gradle:

        - Kotlin Coroutines (`core` и `android`),

        - Jetpack Lifecycle (`viewmodel-ktx`, `livedata-ktx` если нужно, `lifecycle-runtime-ktx`),

        - Room (`runtime`, `ktx` и `compiler` (kapt)),

        - Navigation Component (`fragment-ktx`, `ui-ktx`, Safe Args Gradle plugin),

        - Hilt (`hilt-android` и `hilt-compiler` (kapt)),

        - Material Components (для UI элементов, например, `com.google.android.material:material`),

        - RecyclerView, possibly ConstraintLayout, etc.,

        - (Опционально) Firebase BOM, если планируется легко подключить аналитики или Crashlytics (можно отложить до после MVP).  
          Убедиться, что версии актуальны, и добавить plugin `kotlin-kapt` для kapt (Room, Hilt) в модулях app и data. В `app/build.gradle` включить `viewBinding true`. В `app` и `data` подключить `kotlin-parcelize` если Entity или модели будут Parcelable (например, для передачи объектов через SafeArgs).

    - **Настройка Hilt**: Создать класс Application (например, `XPumpApp : Application`), пометить `@HiltAndroidApp`. В `AndroidManifest.xml` указать этот класс в атрибуте `application android:name`. Создать базовые Hilt-модули: `DatabaseModule`, `DaoModule`, `RepositoryModule` с аннотацией `@InstallIn(SingletonComponent::class)` для предоставления синглтон-зависимостей на уровень всего приложения (БД, репозитории). Пример:

      `@Module @InstallIn(SingletonComponent::class) object DatabaseModule {     @Provides @Singleton     fun provideDatabase(@ApplicationContext context: Context): XPumpDatabase {         return Room.databaseBuilder(context, XPumpDatabase::class.java, "xpump.db").build()     }     @Provides     fun provideWorkoutDao(db: XPumpDatabase): WorkoutDao = db.workoutDao()     // ... аналогично для других DAO }`

      В `RepositoryModule` предоставить реализации, например:

      `@Provides @Singleton fun provideWorkoutRepository(dao: WorkoutDao, userPrefs: PreferencesStorage): WorkoutRepository {     return WorkoutRepositoryImpl(dao, userPrefs) }`

      При необходимости, настроить Hilt для ViewModel: добавить зависимость `androidx.hilt:hilt-lifecycle-viewmodel` (для старых версий) или использовать встроенную аннотацию `@HiltViewModel`. Добавить `kapt "androidx.hilt:hilt-compiler"` если используем hilt-viewmodel.

    - **Навигация**: Добавить Safe Args плагин (`classpath "androidx.navigation:navigation-safe-args-gradle-plugin:версия"`) в проект и apply plugin `'androidx.navigation.safeargs.kotlin'` в module app. Создать nav_graph.xml (можно пока пустой или с одним заглушечным экраном) и настроить MainActivity как NavHost. В `activity_main.xml` добавить `FragmentContainerView` с `navGraph` и `navController`. Проверить запуск пустого проекта на устройстве.

2. **Реализация слоя данных (Room и модели)**  
   Первым делом нужно спроектировать и создать модели данных и таблицы, чтобы на них строить остальной функционал:

    - **Сущности и DAO**: В модуле `data` создать класс `XPumpDatabase` с аннотацией `@Database(entities=[WorkoutEntity::class, ExerciseEntity::class, ProgramEntity::class, WorkoutSetEntity::class, ...], version=1)`. Определить DAO-интерфейсы:

        - `WorkoutDao`: методы `getAllWorkouts(): Flow<List<WorkoutEntity>>` (сортировать по дате), `insertWorkout(workout: WorkoutEntity): Long`, `insertSets(sets: List<WorkoutSetEntity>)`, `getWorkoutWithSets(id): WorkoutWithSets` (можно через `@Transaction` + связь 1->многие).

        - `ExerciseDao`: метод `getAllExercises(): LiveData<List<ExerciseEntity>>` или Flow, `insertAll(List<ExerciseEntity>)` (для первичного наполнения).

        - `ProgramDao`: `getAllPrograms(): List<ProgramEntity>`, `updateProgram(program)` (если надо помечать выполнение/прогресс). Программы могут содержать список упражнений, это может быть реализовано через отдельную таблицу-связку ProgramExercise или хранить в JSON – но для MVP можно упростить, например, иметь поле `exercises: String` с ид упражнений через запятую. Однако лучше правильно спроектировать: возможно, завести таблицу `ProgramDayEntity` и `ProgramExerciseEntity` если планируется детальная структура программ. Для MVP, если программы простые (набор упражнений), можно обойтись хранением их в JSON-формате или отдельной таблицей ProgramExercises.

        - `UserDao` (опционально): если хранить профиль (уровень, опыт) в БД. Но по ТЗ уровень и опыт – в SharedPreferences, так что можно пока не делать UserDao.

    - **Entity классы**:

        - `ExerciseEntity(@PrimaryKey val id: Int, val name: String, val muscleGroup: String, val description: String, val imageRes: Int?)` – сущность упражнения. Желательно предусмотреть поле для ссылки на изображение (например, имя файла в assets).

        - `ProgramEntity(@PrimaryKey val id: Int, val name: String, val levelRequired: Int, val description: String)` – сущность программы тренировок, с полем требуемого уровня для разблокировки. Если программы содержат несколько дней или упражнений, можно расширить схему.

        - `WorkoutEntity(@PrimaryKey(autoGenerate=true) val id: Long = 0, val date: Long, val programId: Int?)` – сущность завершённой тренировки. Хранит дату (в мс) и опционально ссылку на программу, по которой она выполнялась (или null, если произвольная). Можно добавить поле суммарного объёма или длительности, но это не обязательно.

        - `WorkoutSetEntity(@PrimaryKey(autoGenerate=true) val id: Long = 0, val workoutId: Long, val exerciseId: Int, val weight: Float, val reps: Int)` – сущность подхода: ссылается на тренировку и упражнение, хранит вес и повторения. Тут можно добавить поле порядкового номера или просто считать порядок по id.

        - (Если потребуется) `UserEntity` – например, для хранения веса пользователя или рекордов по упражнениям, но можно и без этого, используя SharedPrefs.  
          Проверить, что связи Workout -> WorkoutSet (один-ко-многим) можно получить через `@Relation` или через запрос с JOIN при необходимости.

    - **SharedPreferences**: создать класс `PreferencesStorage` (в модуле data или отдельном модуле common). Он может быть Singleton (например, через Hilt @Singleton) и использовать `@ApplicationContext` для доступа к `context.getSharedPreferences("xpump_prefs", MODE_PRIVATE)`. Методы: `getLevel(): Flow<Int>`, `setLevel(value: Int)`, `getExperience(): Flow<Int>`, `addExperience(xp: Int)` (внутри которого увеличивать опыт и, если превысило порог – увеличить уровень), `getWeightCategory(): String`, `setWeightCategory(cat: String)`, `isFirstRun()` для возможной логики первого запуска (например, чтобы заполнить упражнения в БД). Для реализации Flow можно использовать `callbackFlow` или просто читать из prefs при каждом запросе (так как уровень меняется нечасто, допустимо). DataStore бы позволил получать Flow прямо, но с SharedPrefs придется писать обертку. Пока можно упростить: использовать обычные функции get/set, а во ViewModel вызывать их когда нужно (например, получать уровень при старте профиля).

    - **Инициализация данных**: Рассмотреть, как заполнить справочники (упражнения, программы) при первом запуске. Это можно сделать либо через отдельный use-case, вызываемый при старте приложения, либо через специальную функцию `prepopulateDatabase()`. Простой способ: пометить БД аннотацией `@Database(..., exportSchema=true)` и использовать Room callback `Room.databaseBuilder(...).addCallback()` с заполнением. Но заполнение десятками строк удобнее сделать вручную: например, в `PreferencesStorage` хранить флаг firstRun, и если true – выполнить в IO-потоке вставку дефолтных упражнений и программ. Дефолтные данные можно хранить в ресурсах (JSON файл в assets или raw). На MVP этапе можно захардкодить несколько записей прямо в коде (например, создать List<ExerciseEntity> и вставить). Библиотека упражнений: по условию, должна быть группировка по мышцам и описания – эти данные желательно иметь. Возможно, не все упражнения нужны сразу; можно внести хотя бы 5-10 упражнений для примера. Программы: тоже 2-3 программы с разными levelRequired. Это подготовит почву для геймификации.

3. **Репозитории и UseCase – реализация бизнес-логики**

    - **Имплементация репозиториев**: После создания DAO и моделей можно реализовать классы RepositoryImpl. Они находятся в data-модуле и помечены `@Inject` для конструктора (чтобы Hilt мог их создавать). Примеры:

      `class WorkoutRepositoryImpl @Inject constructor(     private val workoutDao: WorkoutDao,     private val exerciseDao: ExerciseDao,     private val prefs: PreferencesStorage ) : WorkoutRepository {     override fun getAllWorkouts(): Flow<List<Workout>> {         return workoutDao.getAllWorkouts()             .map { list -> // при необходимости маппим Entity -> Domain                 list.map { it.toDomain(exerciseDao) }              }     }     override suspend fun addWorkout(workout: Workout, sets: List<WorkoutSet>) {         // Вставляем тренировку и связанные подходы в транзакции         val workoutId = workoutDao.insertWorkout(workout.toEntity())         sets.forEach { set -> workoutDao.insertSet(set.toEntity(workoutId)) }         // Начисляем опыт и сохраняем через Preferences         prefs.addExperience(calculateXpForWorkout(workout, sets))     }     // ... другие методы }`

      Здесь показано, как репозиторий может совмещать логику: сохранять данные в БД и обновлять профиль. `calculateXpForWorkout` – некоторая функция (может быть в UseCase) для вычисления опыта за тренировку, например 10 xp за тренировку + 1 xp за каждый подход или зависящая от поднятого веса. На MVP можно сделать просто фиксированное значение или суммировать все повторения. Аналогично, `ProgramRepositoryImpl` может получать список всех программ из БД, и помечать их разблокированность:

      `override fun getPrograms(): Flow<List<Program>> {     return programDao.getAll().map { list ->         val level = prefs.getLevel()         list.map { it.toDomain(isUnlocked = level >= it.levelRequired) }     } }`

      `ExerciseRepositoryImpl` – просто возвращает список упражнений из DAO (или из памяти, если хотим кэшировать). `UserRepositoryImpl` – если делаем, будет обёрткой над PreferencesStorage (например, `getLevel()` вызывает prefs.getLevel()).  
      Обратите внимание, что **большая часть кода репозиториев может быть сгенерирована или упрощена**: Room сгенерирует SQL доступ, Hilt – провайдеры. Задача разработчика – написать логику, специфичную для приложения (начисление опыта, фильтрация). Мэпперы `toDomain()`/`toEntity()` можно сгенерировать с помощью data class copy или написать вручную (поскольку наши модели похожи, можно даже использовать одни и те же классы, но тогда domain будет зависеть от Room entity, что не по Clean Architecture – лучше разделить модели).

    - **Написание UseCase (Interactors)**: Решить, нужны ли отдельные use-case классы или достаточно вызывать репозитории из ViewModel. В небольшом приложении можно обойтись без явных UseCase-классов, но для демонстрации архитектуры и возможного роста лучше их внедрить. Например:

        - `LogWorkoutUseCase @Inject constructor(private val workoutRepo: WorkoutRepository, private val userRepo: UserRepository)` с функцией `invoke(workout: Workout, sets: List<WorkoutSet>)` – внутри вызывает `workoutRepo.addWorkout(...)` и, возможно, возвращает результат (успешно или бросает исключение).

        - `GetWorkoutHistoryUseCase(private val workoutRepo)` возвращает Flow<List<Workout>> через `workoutRepo.getAllWorkouts()`.

        - `GetProfileOverviewUseCase(private val workoutRepo, private val userRepo, private val normsRepo)` – собирает данные для экрана профиля: текущий уровень/опыт (из userRepo), лучшие результаты из workoutRepo (например, запрос максимального веса по каждому ключевому упражнению – для этого можно добавить метод `getBestResult(exerciseId)` в WorkoutDao), и возвращает объект, содержащий всю информацию (уровень, опыт, список ProgressBarModel для каждого упражнения с текущим результатом и процентом до следующего разряда). Этот use-case как раз инкапсулирует вычисление нормативов: он может воспользоваться статическими таблицами нормативов. Например, хранить в коде `val benchPressNorms = mapOf("Class3" to 100, "Class2" to 120, ... )` или формулой. С учетом веса пользователя (который вводится в профиле) можно выбрать норматив. Хотя MVP может упростить и задать фиксированные нормы независимо от веса, архитектура должна позволять учитывать вес – можно хранить его в UserPreferences и учесть при расчёте.

        - `AwardExperienceUseCase(private val userRepo)` – увеличить опыт, проверить повышение уровня: (например, если опыт > threshold, level++). Threshold можно задать формулой или константами (e.g., 100 xp per level). Вызывать этот use-case при добавлении тренировки.

        - **Генерация UseCase**: код use-case не генерируется автоматически, но писать его удобно шаблонно. Можно воспользоваться Live Templates в Android Studio, или просто следовать одному стилю (например, UseCase с оператором invoke, или публичным методом execute).

        - **Связывание UseCase с ViewModel**: через Hilt мы можем инжектить репозитории напрямую в ViewModel или инжектить UseCase. Как более чистый подход, создадим UseCase и инжектим их. Например, `ProfileViewModel @HiltViewModel constructor(private val getProfileUseCase: GetProfileOverviewUseCase, ...)`. Это немного увеличивает количество классов, но повышает **тестируемость** и повторное использование.

4. **Реализация экранов (UI) и навигации**  
   Далее, приступить к созданию фрагментов, их макетов и привязки к ViewModel. Рекомендуемый порядок – сначала сделать ключевые экраны, необходимые для подтверждения работы данных:

    - **Экран профиля (ProfileFragment)** – как центральный экран, показывающий суммарную информацию: уровень пользователя, прогресс XP (например, текст “XP: 40/100 до следующего уровня”), прогресс-бары нормативов по 3-4 упражнениям (жим, присед, тяга, и возможно армейский жим). Этот экран требует данных из разных источников (SharedPrefs для уровня/опыта, Room для лучших результатов, статических нормативов для расчета). Поэтому начинать с него рационально после того, как будут рабочие репозитории/use-case. Разметка: использовать ConstraintLayout или NestedScrollView; в вершине – блок профиля (аватар заглушка, имя – если введём, можно просто “User”), уровень и опыт (TextView), прогресс-бар уровня (можно Material ProgressBar, второстепенный; либо CircularProgressIndicator). Ниже – список ключевых упражнений и прогресс до следующего разряда: можно сделать в виде повторяющегося элемента (иконка упражнения, текст текущий вес и целевой норматив, графический прогресс). Реализовать это можно через RecyclerView горизонтальный или просто линейно, если их фиксированно 3-4. Под списком – кнопка “Библиотека упражнений” и “Программы” для навигации (или вынести их в нижнее меню).

        - **ProfileViewModel**: вызывается `getProfileOverviewUseCase` при старте (в `init` или при `onViewCreated`). Возвращает профильные данные (можно использовать `liveData` builder или StateFlow). ProfileViewModel преобразует Domain-данные в UI-модель (например, строку “Вы жмёте 80 кг, до КМС нужно 100 кг”). Также обрабатывает нажатия, например, кнопку “Начать тренировку” (если её сделаем на профиле) – тогда отправляет навигационное событие (можно через интерфейс, но лучше просто вызывать navController из Fragment по клику).

    - **Экран начала тренировки / выполнения** (`WorkoutSessionFragment`) – важная часть трекера. Он позволяет пользователю выбрать упражнения и ввести свои подходы. Возможны варианты: либо пользователь выбирает одну из разблокированных программ (тогда упражнения предопределены), либо составляет тренировку сам (выбирает упражнения из списка). Для MVP можно реализовать упрощённо: если запуск через Program, то на экране показываются упражнения из этой программы; если запуск свободной тренировки, то дать пользователю + button “Добавить упражнение” (показывает диалог выбора упражнения из списка). Основной функционал – ввод подходов: скорее всего, сделать динамический список, где для каждого упражнения отображается список его подходов (например, текстовые поля для веса и повторений). Можно реализовать это так: завести внутреннюю модель `SetInput(exerciseId, weight, reps)`. В ViewModel хранить список таких SetInput. Иметь кнопку “Добавить подход” под каждым упражнением. Для упрощения UI можно не делать группировку по упражнениям, а просто линейный список всех введенных подходов (но это менее наглядно). Лучше группировать: использовать RecyclerView с несколькими viewType (Exercise header + list of sets). Это достаточно сложно для MVP, поэтому допускается упрощение: например, реализовать ввод подходов для **одного упражнения за раз** – пользователь выбирает упражнение, вводит последовательность подходов для него (можно через диалог), сохраняет, потом выбирает следующее. Однако, так UX страдает. Возможно, лучше первично реализовать просто ввод одного подхода (упр+вес+повторы) – нажал “Сохранить подход”, он уходит в БД, остается на экране. Повторить для следующего подхода. По завершении – кнопка “Завершить тренировку”.

        - **WorkoutSessionViewModel**: содержит либо выбранную программу (если передана через SafeArgs), либо флаг свободного режима. Держит список выбранных упражнений/подходов (например, Map<Exercise, List<SetInput>>). При каждом добавлении подхода обновляет этот список (можно через MutableStateFlow). Команда “Завершить” вызывает `LogWorkoutUseCase` – передавая собранные данные. UseCase сохранит в БД и начислит опыт. После этого можно навигировать назад на профиль или историю.

        - Разметка: несколько TextView/Spinner для выбора упражнения (если свободный режим), RecyclerView для списка добавленных подходов, кнопки “+Подход” и “Завершить”. Для ввода веса/повторов можно использовать TextInputEditText в диалоге либо сразу в элементах списка (но тогда нужно обеспечить сохранение при прокрутке – можно держать EditText привязанные к LiveData, но это сложнее; проще через диалог).

    - **Экран истории (WorkoutHistoryFragment)** – отображает список завершённых тренировок (с датами и краткой информацией). Реализация: простой вертикальный RecyclerView. Элемент списка содержит дату (форматировать в человекочитаемый вид), название программы или отметку “Свободная тренировка” если programId = null, и например количество упражнений/подходов в ней (для краткости). Можно также сразу показывать заработанный опыт за тренировку, если есть смысл. Для красоты – иконку или emoji для разных программ.

        - **HistoryViewModel**: при инициализации запрашивает Flow истории: `getWorkoutHistoryUseCase()`, преобразует в LiveData/StateFlow. Также может иметь функцию `onWorkoutSelected(id)` – если решим открывать детали по тапу (в MVP можно не делать экран деталей, а просто неактивный список). Репозиторий возвращает Flow, так что список обновится автоматически при добавлении новой тренировки.

        - Макет: `RecyclerView` внутри SwipeRefreshLayout (если добавить pull-to-refresh для сетевой синхры, но пока не нужно – история локальная). Адаптер — `HistoryAdapter` на базе ListAdapter<Workout, VH>. DiffUtil сравнивает по id. При клике можно сделать Toast “Тренировка X завершена”, либо реализовать переход на подробный экран (в MVP можно опустить подробный экран, т.к. основные данные истории уже на главном профиле показываются, а детали – список подходов – можно добавить позже).

    - **Экран списка упражнений (ExerciseListFragment)** – показывает библиотеку упражнений, сгруппированную по группам мышц. Реализовать можно через RecyclerView с SectionHeaders: либо использовать две колонки (RecyclerView + ItemDecoration для разделителей), либо простой список, где элементы – это CardView с названием упражнения и подписью (группа мышц). Группировку можно визуально обозначить (например, отсортировать по muscleGroup и выводить заголовок группы как ненажимable item). MVP-реализация: просто отсортировать и, при смене группы, вставлять элемент-заголовок (нужно чуть более сложный адаптер). Либо проще: сделать Spinner сверху “Фильтр по группе” для выбора группы мышц, по умолчанию “Все”. Это сильно упростит: при выборе группы – фильтровать список.

        - **ExerciseViewModel**: запрашивает список упражнений из `exerciseRepository.getAllExercises()` (можно сразу LiveData из DAO или Flow, т.к. библиотека статична после заполнения). Держит список и текущий фильтр. Фильтрацию делает в ViewModel (получив весь список, применяет `.filter` по группе).

        - По нажатию на упражнение можно открывать подробное описание (ExerciseDetailFragment) – там показать описание техники и фото. Если фотоинструкции хранятся (например, имя изображения), можно положить картинки в `drawable/` и хранить id (R.drawable.name) или uri. В MVP достаточно статичного текста.

    - **Экран программ (ProgramListFragment)** – показывает доступные тренировочные программы. Представление: похожее на список упражнений, но у каждой программы можно отобразить уровень, с которого она доступна. Если программа заблокирована (userLevel < levelRequired), элемент может быть тусклым и не кликабельным, с замочком. Если разблокирована – по нажатию стартует WorkoutSessionFragment с этой программой.

        - **ProgramViewModel**: запрашивает `getPrograms()` из репо. Можно сразу возвращать List<Program> с флагом `isUnlocked`, как мы планировали. Далее – бизнес-логика отображения: можно сортировать по levelRequired, или разделить на “доступные/недоступные”.

        - Адаптер ProgramAdapter: в методе onBind проверяет program.isUnlocked, меняет background или alpha. При клике: если unlocked – навигировать к началу тренировки (передав programId), если locked – можно показать Toast “Требуется уровень X”.

    - **Экран настроек** – опционально, если нужны переключатели (например, единицы измерения кг/фунты, уведомления on/off). MVP может обойтись без отдельного экрана настроек, но хотя бы управление весовой категорией пользователя нужно где-то сделать. Возможно, это делается на экране профиля: например, кнопка “Изменить вес” открывает диалог ввода веса тела, и результат сохраняется в SharedPreferences. Так будет проще, чем городить целый экран настроек. Другие настройки (уведомления, темы и т.п.) не критичны для MVP.


    Реализуя каждый экран, сначала создаём макет XML, затем Fragment класс, привязываем ViewModel. Во фрагменте через `by viewModels()` (Hilt автоматически предоставит фабрику) получаем ViewModel. В `onViewCreated` устанавливаем observers на LiveData/StateFlow из ViewModel, обновляем адаптеры/виджеты. Также устанавливаем слушатели на кнопки: например, `binding.startWorkoutButton.setOnClickListener { findNavController().navigate(... к WorkoutSessionFragment ...) }`. Можно использовать Navigation SafeArgs для передачи данных (например, Program ID). В ViewModel при надобности вызываем UseCase в фоне: либо через `viewModelScope.launch` (если `suspend` функции), либо `.asLiveData()` для Flow. Обработку ошибок (например, Room может бросить исключение) стоит тоже учесть: try-catch в ViewModel и запись в LiveData `errorMessage` для показа Toast.
    
    **Порядок реализации экранов**:
    
    1. Профиль – чтобы сразу просматривать основную информацию и отладить логику опыта/уровней.
        
    2. История – чтобы видеть сохраняются ли тренировки (можно сначала вручную добавить несколько записей в БД для теста).
        
    3. Экран выполнения тренировки – наиболее интерактивная часть (здесь можно начать с упрощённого варианта: выбор 1 упражнения и ввод 1 подхода, сохранить – чтобы протестировать цепочку добавления). Затем усложнять до полного сценария.
        
    4. Библиотека упражнений – можно реализовать, как менее приоритетно, после убеждения, что основная логика работает.
        
    5. Программы – тоже после упражнений, т.к. зависят от них. Впрочем, можно и раньше, но они требуют, чтобы профиль показывал уровень для разблокировки.
        
    6. Детали упражнений – по желанию, можно добавить в самом конце как улучшающую фичу.
        
    7. Настройки/диалоги ввода – дополняются по необходимости.
        
    
    После создания каждого экрана зарегистрировать его в `nav_graph.xml` с уникальным id и настроить переходы (action). Проверить навигацию (можно запустить app, использовать Navigation Preview).

5. **Написание кода вручную vs автогенерация**:  
   Проект XPump использует ряд технологий, которые избавляют разработчика от шаблонного кода:

    - **ViewBinding** – автогенерирует классы привязки для layout, не нужно писать `findViewById`. Достаточно включить флаг в Gradle и использовать.

    - **Room** – генерирует большую часть кода доступа к БД. Разработчик описывает только интерфейсы DAO и Entity-классы, все CRUD-методы (SQL синтаксис) обрабатываются компилятором. Например, DAO методы с запросами превращаются в готовые реализации в runtime (через аннотации). Нужно быть внимательным с миграциями, но для версии 1 можно поставить `.fallbackToDestructiveMigration()` пока.

    - **Hilt (Dagger)** – генерирует код для графа объектов. Не нужно вручную писать фабрики ViewModel или синглтон классы – достаточно аннотаций `@Inject` и `@Module`. Hilt также создает класс `XPumpApp_HiltComponents` и пр., скрывая сложности Dagger.

    - **Navigation SafeArgs** – генерирует типобезопасные классы аргументов. Например, если мы указываем в графе, что `WorkoutSessionFragment` принимает `programId: Int`, плагин сгенерирует класс `WorkoutSessionFragmentArgs` и метод `Directions`. Мы можем писать `findNavController().navigate(ProfileFragmentDirections.actionProfileToWorkout(programId = 5))` вместо составления Bundle вручную. Это уменьшает вероятность ошибок и улучшает читаемость.

    - **Data Binding** (не используется явно, мы применяем ViewBinding) – но если бы нужно было сделать сложное отображение списков или логики в XML, можно подключить. В нашем случае это избыточно.

    - **Корутинные утилиты**: использование `viewModelScope.launch` и Flow API устраняет необходимость вручную писать потоки для фоновых задач (в старом AsyncTask или Handler).

    - **Android Studio** также предоставляет **Live Templates и Code Generation**: можно быстро создать класс ViewModel, выбрав шаблон (если настроить), или даже автогенерацию data class из JSON (плагин). Для нашего MVP, скорее всего, ручное написание моделей – это нормально, но инструментами можно ускорить (например, сформировать data классы Exercise по заранее имеющемуся JSON-списку).


    В конечном счёте, ручному написанию подлежат только **уникальные части логики**: расчёты нормативов, бизнес-правила начисления опыта, конфигурация UI. Все шаблонные элементы – навигация, доступ к данным, DI связки – за нас делают библиотеки при минимальной декларации (аннотации/XML). Это позволяет сконцентрироваться на правильности реализации фитнес-логики, а не на рутинной обвязке.

6. **Тестирование (unit, UI, интеграционное)**  
   С самого начала стоит планировать тесты, поскольку архитектура специально способствует тестируемости. Ключевые уровни тестирования:

    - **Юнит-тесты бизнес-логики**: для domain-слоя. Можно написать тесты на UseCase. Например, протестировать `AwardExperienceUseCase`: при разных входных данных (XP до, XP начислить) правильно ли повышается уровень и остаётся ли корректным остаток XP. Поскольку UseCase – обычный Kotlin класс без Android, использовать JUnit4/5 и Assertions. Для изоляции можно замокать репозитории (например, с Mockito или писать фейковые реализации). Domain-слой изначально не зависит от Android, поэтому тесты быстрые.

    - **Юнит-тесты репозиториев**: здесь можно использовать инструментальные тесты (AndroidJUnit4) либо mokito. Репозиторий WorkoutRepositoryImpl, например, можно протестировать с использованием In-Memory Room БД, чтобы проверить корректность операций. Room предоставляет `Room.inMemoryDatabaseBuilder(Context, ...).build()` – его можно вызвать в тесте (используя например `ApplicationProvider.getApplicationContext()`). Затем вручную вставить данные через DAO и вызвать методы репозитория, проверяя, что выход соответствует ожиданию. Можно также использовать библиотеку `androidx.arch.core:core-testing` для тестирования LiveData (есть утилита InstantTaskExecutorRule).

    - **Инструментальные тесты UI**: с фреймворком **Espresso** можно автоматизировать сценарии. Например, написать тест, который запускает приложение, навигирует на экран добавления тренировки, вводит данные (в EditText вес и повторы), нажимает “Сохранить” и проверяет, что в истории появился новый элемент. Для этого, возможно, придётся использовать `ActivityScenario.launch()` в тесте и ViewMatchers для взаимодействия с UI. Espresso хорошо работает, если UI использует id у элементов. Не забыть включить `androidx.test.ext:junit` и `androidx.test.espresso:espresso-core` в зависимости `androidTestImplementation`.

    - **Mocking и DI в тестах**: Hilt предоставляет механизм для тестирования с подменой зависимостей – @HiltAndroidTest и создание тестовых модулей (с аннотацией @TestInstallIn) для, например, замены реальной БД на in-memory, или репозитория – на фейковый. Можно этим воспользоваться для UI тестов: например, чтобы не использовать настоящую БД (хотя in-memory тоже ок). На этапе MVP можно часть сложных тестов опустить, но базовые стоит реализовать, чтобы регрессии не ломали функционал.

    - **Производительность и memory leaks**: с таким подходом (корутины, ViewModel) утечек памяти быть не должно при правильном использовании (ViewModelScope автоматически отменяется при уничтожении). Однако, можно написать тест на повороты экрана: запустить Fragment, повернуть (ActivityScenario recreate), убедиться, что данные ViewModel сохранились (например, введенные подходы не потерялись).

    - **Тестирование граничных случаев**: проверить, что будет, если у пользователя 0 упражнений (пустой список – UI должен отобразить например текст “Нет данных”), если добавить тренировку без подходов (предотвратить это на UI – кнопка “Завершить” не активна, пока нет ни одного подхода), длинные названия упражнений – не ломают ли верстку и т.п.

    - **Линтеры и статический анализ**: подключить detekt, ktlint для поддержания чистоты кода, а также профилировать приложение (Android Profiler) на предмет затратных операций – но на MVP с локальной БД всё должно работать быстро.

Следуя этому плану, команда разработчиков сможет сначала создать прочный **каркас приложения**, а затем наполнять его конкретной логикой фитнес-трекера.
